# Ring-3 Linux Process Host

Puck.BareMetal provides the Linux ABI needed to run one engine process directly
from a UEFI image. It is a purpose-built host, not a general Linux kernel: there
is one address space, one process, a bounded thread table, and no `fork` or
`exec`.

## Execution model

`samples/EfiLinux` performs the complete boot flow:

1. UEFI resources and staged files are captured before `ExitBootServices`.
2. The native kernel selects conventional memory, initializes its allocator,
   installs a GDT and IDT, creates page tables, and switches `CR3`.
3. The kernel initializes PAT mappings, the framebuffer console, PCIe ECAM,
   clock, interrupts, DMA, networking, and the AMD GPU path when compatible
   hardware is present.
4. `Puck.Runtime` initializes reference statics and invokes managed `Main`.
5. The managed ELF loader maps a musl x86-64 executable and its dynamic
   dependencies, builds the System V entry stack, and enters ring 3.
6. Guest system calls cross the `SYSCALL` trampoline into the kernel dispatcher.

Guest pages are user-accessible; kernel pages remain supervisor-only. CPU faults
enter the IDT panic path and print a trap frame instead of silently resetting the
machine. The PIT preempts guest threads at approximately 100 Hz, while kernel code
runs with interrupts disabled.

## ELF loading

The loader supports static and dynamic musl PIE executables:

- `PT_LOAD` mapping, BSS zeroing, and load-bias relocation;
- `PT_INTERP`, `AT_BASE`, `AT_ENTRY`, `AT_PHDR`, and the other auxiliary values
  needed by `ld-musl`;
- a System V `argc`/`argv`/`envp`/`auxv` entry stack;
- file-backed mappings for staged shared objects;
- distinct device and inode identities so the dynamic loader does not collapse
  unrelated libraries;
- canonical path and symbolic-link resolution for `/dev` and `/sys` probes.

The RADV closure is too large to embed in the EFI image. It is copied from the
ESP before firmware services are released and exposed through the synthetic VFS.

## Supported guest services

The syscall dispatcher implements the surface exercised by musl, the dynamic
loader, RADV, and the conformance guests. The main groups are:

- process and memory: `mmap`, `mprotect`, `munmap`, `mremap`, `madvise`, `brk`,
  `arch_prctl`, `exit`, and `exit_group`;
- threads: `clone`, `futex`, `sched_yield`, `gettid`, `set_tid_address`, and
  `set_robust_list`;
- files: `open`, `openat`, `close`, `read`, `pread64`, `readv`, `write`,
  `writev`, `lseek`, `fcntl`, `stat` variants, `getdents64`, and `readlink`
  variants;
- platform: `clock_gettime`, `getrandom`, affinity and identity queries, and
  the signal calls that musl probes;
- devices: DRM and AMDGPU ioctls on `/dev/dri/renderD128`.

Unsupported calls return `-ENOSYS` and log the syscall number. Several supported
calls intentionally provide the narrow semantics required by this single-process
host: mappings come from bump arenas, signal delivery is not modeled, and
`exit_group` persists diagnostics and parks the kernel.

## Threads and scheduling

Each guest thread owns a full saved register context and FS base. `clone` creates
a context in a bounded kernel table; `futex` supplies wait/wake behavior for musl
pthreads; `exit` clears the child-TID address and wakes joiners. Both syscall
return and timer preemption resume through the same `iretq` path. Scheduling is
round-robin over runnable threads.

This is thread support within one process. There is no process isolation,
multi-process scheduler, SMP, or overcommit policy.

## Files, networking, and TLS

The synthetic read-only VFS exposes:

- process and device files required by musl;
- `/dev/dri/renderD128` and the corresponding `/sys` topology;
- staged RADV and musl shared objects;
- small diagnostic resources such as `/proc/cpuinfo`, `/proc/meminfo`,
  `/dev/null`, `/dev/zero`, and `/dev/urandom`.

Networking uses a transitional virtio-net device and vendored lwIP in `NO_SYS`
mode. The kernel supports DHCP, DNS, TCP, and an HTTPS client. TLS is provided by
vendored mbedTLS through the `PuckTls` abstraction, with hardware entropy,
real-time clock support, hostname verification, and an embedded CA subset. The CA
bundle is regenerated by
`compat/native/mbedtls-port/regen-ca-bundle.ps1`; its provenance belongs in
`NOTICE.md` and the generated source, not in operational logs.

## DRM and RADV boundary

The render node implements the kernel-facing shape needed for RADV discovery and
shader compilation: DRM version/capability queries, AMDGPU device and memory
information, GEM allocation and mapping, context handles, synchronization-object
stubs, and a host-backed BO table. Under QEMU this supports Vulkan instance and
device creation, resource allocation, and ACO compilation to RDNA2 ISA.

QEMU does not emulate the target RDNA GPU. Real GPU virtual memory, command
submission, and fence completion are exercised only on Steam Deck hardware. See
[AMD Vulkan Host](amd-vulkan-plan.md) and
[Steam Deck GPU Bring-up](deck-bringup-handoff.md).

## Build and verification

Run from a Visual Studio Developer PowerShell with QEMU and OVMF installed:

```powershell
samples/EfiLinux/run-qemu.ps1
```

The script performs a clean publish because the NativeAOT link target does not
track every native object as an incremental input. It stages an ESP and boots the
image headlessly with the serial log captured to disk.

Useful variants:

```powershell
samples/EfiLinux/run-qemu.ps1 -Pxe
samples/EfiLinux/stage-deck.ps1 -Target E:\
```

A successful QEMU run reaches the guest, completes the network/TLS probe when
network access is available, enumerates the synthetic RADV device, compiles the
compute pipeline with ACO, and exits through `exit_group(0)`. Network or host-tool
availability must be reported separately from kernel failures.

## Current constraints

- One process and one address space; no `fork`, `exec`, SMP, or general signal
  delivery.
- Page allocation and guest mappings are intentionally simple and are not
  reclaimed.
- `exit_group` parks instead of returning to managed `Main`.
- The CA bundle is a curated subset, not a general operating-system trust store.
- The target Deck does not expose an IVRS table, so the current path uses direct
  DMA. An IOMMU domain is required before supporting hardware that enables AMD-Vi.
- GPU command execution and fence completion require the Deck; QEMU proves only
  the host ABI and userspace-driver ceiling.
