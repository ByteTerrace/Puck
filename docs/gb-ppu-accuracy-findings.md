## Root cause

The PPU runs a **per-dot BG fetcher** (`Ppu.Fetcher.cs`) **alongside a closed-form mode-3 length**
(`Ppu.cs`: `m_mode0StartDot = 80 + DrawingDots + ScxPenalty + ObjectPenalty`). The two are **decoupled** â€” the
fetcher emits pixels per dot, but the mode-0 transition (and thus the STAT/HBlank interrupt) fires at the
independently-computed dot. The header comment in `Ppu.Fetcher.cs` already flags this as "Phase 1": *"Sprites are
still overlaid at line end and the closed-form mode-3 length still drives timing; both are folded into the fetcher
in later phases."*

Consequences:
1. **First-pixel alignment is wrong.** The fetcher emits its first pixel at mode-3 dot â‰ˆ8; hardware emits at dot 12
   (the 172-dot minimum mode-3 = 12 warmup + 160 pixels). So mid-line register writes (BGP/SCX/LCDC/OBP) land ~4
   pixels off from the warmup alone.
2. **CPU/PPU phase contributes the rest.** The remaining ~8 px of the +12 is a phase offset between when a CPU
   register write becomes visible to the fetcher and the dot the hardware applies it. This is entangled with the
   deferred-cycle bus model (which is correct for the timer â€” TimerDiv 98%).
3. **Mode-0 interrupt is closed-form, not fetcher-driven**, so SCX-dependent sub-dot cases (gbmicrotest
   `hblank_int_scx*`) are Â±1 off even though the coarser mooneye `intr_2_mode0_timing` / `hblank_ly_scx_timing`
   pass.

## Why it can't be fixed by tweaking a constant

The emulator is already tuned so the **mode transitions** match mooneye's resolution (those tests pass). Nudging
`DrawingDots`/`ScxPenalty`/the warmup by N to satisfy gbmicrotest or mealybug **regresses** the passing mooneye PPU
and the mode-timing-adjacent suites. The error is sub-dot and structural, not a single miscalibrated value.

## Refactor progress (the FIFO-driven decoupling â€” landed)

The structural fix has been implemented for the DMG BG path and is **zero-regression** (blargg/mooneye/timer/
scribbl/acid2 all still green; full scorecard unchanged at 502/224/34):

1. **Decoupled the pixel pipeline from the closed-form mode-0 transition.** The fetcher now runs on its own clock
   until all 160 pixels are output (`m_bgFetchActive`, `CompleteBackgroundLine`), while `m_mode0StartDot` still
   drives the CPU-visible STAT timing. This is the key insight: hardware's polled-STAT mode-0 (172 dots, what mooneye
   measures) and the pixel pipeline are *different clocks*.
2. **Mode-3 warmup** (`BackgroundWarmupDots = 12`): first pixel at the hardware-faithful dot, so mid-mode-3 register
   writes land on the right pixel.
3. **Sub-dot BGP latch** (`m_bgpRender` + dot-phase deferral in `WriteRegister`): a BGP write landing on a FIFO
   push-dot (`m_dot & 7 == 0`) reaches the pixel pipeline one pop later than a mid-fetch write â€” the exact behavior
   mealybug `m3_bgp_change` encodes.

**Result on `m3_bgp_change` (DMG): pixel error 5804 â†’ 470 (âˆ’92%)**, with most scanlines (incl. rows 1â€“15) now
**pixel-perfect**. But the test needs pixel-EXACT (0 diff) to pass, and the residual 470 is two finer behaviors:
- **Row 0** (~22 px): the first visible line's mode-2-interruptâ†’handler phase differs from steady-state lines.
- **Tile-row-2+ rows** (~4 px each): 1-pixel shade *spikes* at band edges where the tile's colour index interacts
  with the in-flight BGP value â€” an even finer sub-dot latch case.

And each *other* mealybug test (m3_scx, m3_lcdc_*, m3_obp0, the obj tests) probes a *different* register/interaction

rendered output is now ~92% closer to hardware. Pixel-exact mealybug is genuinely multi-month frontier work; the
architecture and the primary BGP latch are now correct and are the foundation for it.

## The real fix (remaining)

Unify the fetcher with mode-3 timing so the **mode-0 transition emerges from the fetcher** rather than a closed
form:
- Model the initial fetch **warmup** (first BG fetch discarded; first pixel at dot 12 for SCX=0).
- Apply the **SCX fine-scroll discard** by dropping pixels from the FIFO front (already partially done via
  `m_scxDiscard`) so its mode-3 lengthening *emerges* instead of being added as `ScxPenalty`.
- Fold the **sprite fetch stalls** into the fetcher (replacing `ObjectPenalty` + line-end sprite overlay) so OBJ
  mid-line effects (mealybug `m3_lcdc_obj_*`) render correctly.
- Drive **mode 0 / the HBlank STAT interrupt** off "the fetcher emitted pixel 160", not `m_mode0StartDot`.

This is the "later phases" the code defers. It is regression-gated by the harness: re-run the `ScorecardFact` and
require blargg/mooneye/timer to stay green while mealybug + gbmicrotest `hblank_int` climb.
