# Disposal implementation audit — 2026-07-17

Full-repo audit of every type that participates in disposal. Written to be actionable by an agent with no other context: every finding carries exact `file:line` evidence, a concrete failure scenario, and an executable action. The refuted-suspicions section is equally load-bearing — it records what was investigated and cleared, so future audits don't re-flag it.

## Method

1. **Discovery (Roslyn, deterministic).** A syntax-level Roslyn scan over all `.cs` files (excluding `obj/`/`bin/`) found **191 types** declaring `Dispose()`/`DisposeAsync`/`Dispose(bool)`/a finalizer or listing `IDisposable`/`IAsyncDisposable` in their base list. Grep was not used for discovery (see `.agents/skills/roslyn-first-analysis`).
2. **Parse + usage (19 Sonnet 5 agents).** Each batch agent read every implementation (guard, finalizer, `GC.SuppressFinalize`, thread-safety, exactly which fields are released) and enumerated every creation site with how disposal happens there (`using`, explicit, owner-field, pooled, never).
3. **Adversarial verification (19 Fable agents).** Each analysis agent re-read the code for every suspected gap, refuted false alarms (external ownership, process-lifetime singletons, deliberate leak-on-exit), independently re-checked the risky types (native/GPU handle owners, worker threads, sessions, pools), and ranked severity against the engine's reality: the demo runs indefinitely and creates/destroys machines, cabinets, and feeds at runtime, so per-session leaks are serious; leak-on-exit in short-lived Post processes is not.

38 agents, all 19 batches completed, 0 errors. Result: **55 findings** (2 high, 7 medium, 34 low, 12 info), **146 refuted suspicions**, **188 clean-type confirmations**.

## Triage: findings clustered into work items

Several findings are one defect observed from multiple batches. An agent picking up this report should work the clusters, not the raw list. Finding IDs below are per-severity (`HIGH-n`, `MEDIUM-n`, `LOW-n`, `INFO-n`) matching the numbered sections that follow. Ordered by priority:

**T1 — Surface-upload/import rebuild paths destroy GPU resources without draining consumers** (HIGH-1, HIGH-2, MEDIUM-5 `VulkanSurfaceImport`, MEDIUM-7 duplicate sighting, LOW-33 `IsDisposed` branch gap, LOW-34 `SurfaceCompositor` cached upload/import surviving a device change). One fix pattern across both backends: before the rebuild-path `DisposeImageResources()`/`DisposeResources()`, drain the way the respective `Dispose()` already does (`WaitForGpu()` on D3D12, `TryWaitIdle()` on Vulkan). Rebuilds fire only on extent/format/device changes, so the stall is rare. This is live-path: a resized WGC capture window or a dmg↔agb device swap (160×144 → 240×160) triggers it in a running demo.

**T2 — Device-lost recovery never reaches OverworldFrameSource-owned GPU objects** (MEDIUM-1 bake preview, MEDIUM-2 diegetic feeds, MEDIUM-4 the root cause). The demo's device-lost notification dies at the `DiegeticUiDirector` decorator's default no-op, so the ViewStack's offscreen engines, face/console feed uploads, and the bake-preview rasterizer all keep resources of the destroyed device and resubmit on it after rebuild. One seam, one fix: forward device-lost through the decorator into `OverworldFrameSource` (reset feeds + `DisposeBakePreview`), mirroring `AdvancedGamingBrickHost.cs:262-266`.

**T3 — Blocking readback races its own pipelined SubmitRead on both backends** (MEDIUM-6 `VulkanSurfaceReadback`, LOW-26 same seam via `SdfWorldEngine.ReadPixels`, LOW-9 `DirectXGpuSurfaceReadback`). `Read()` after an un-consumed `SubmitRead` re-records a pending command buffer, and on extent change destroys the in-flight fence/buffer. Add the in-flight guard every other `SdfWorldEngine` submit path already has.

**T4 — Pooled-fork double-dispose aliases machine state; worker Load-after-Dispose leaks a thread** (LOW-2/3/4 AGB side, LOW-16/17 HGB side; `QueuedMachineWorker` itself lives once in `src/Puck.Hosting`, flagged from both). Double-Dispose parks the same instance twice → two later `Fork()` calls rent one machine to two owners, silently aliasing emulated state. Fix once in each pool's park path (idempotence guard) and once in `QueuedMachineWorker.Load` (disposed check).

**T5 — Constructor exception-safety in native wrappers** (LOW-12/13 DirectX, LOW-21 `Win32D3D11VideoDevice`, LOW-24 `Win32NativeWindow` GCHandle, LOW-27/28/29/30 Vulkan). A throw mid-constructor leaks every already-created native resource because no owner exists yet. Mechanical fix per type: try/catch-cleanup-rethrow, or create into locals and assign at the end. Worth one sweep-style pass over the listed constructors.

**T6 — Non-idempotent or unguarded Dispose** (LOW-11 `DirectXGpuShaderModule` throws on second call, LOW-14 `BufferedConsoleOutput` throws, INFO-4 `OverworldRenderNode` — which IS double-disposed on every normal shutdown and survives only because every callee happens to be idempotent). Cheap guards.

**T7 — Teardown races on thread-join timeout** (LOW-19 `GamepadDevice`, LOW-15 `ProbeWindow`, INFO-10 `Win32XboxAcquisitionSource`, LOW-23 `GameInputHaptics` — the last can crash the process via a freed marshaling thunk when unregistration fails, the most dangerous of the group). Real but low-probability shutdown races.

**T8 — Dead code with correct Dispose implementations** (LOW-5 + INFO-1 capture-tap pipeline `CaptureSink`/`CapturingRenderNode`/`ICaptureSink`; INFO-12 `VulkanGpuExportableRenderTarget` + surface-import wrapper never constructed). Per supergreen doctrine the default is delete-or-wire-up, but the dormant camera GPU zero-copy tier is an owner-declared deliberate keep — **confirm with the owner before deleting anything in this cluster**.

**T9 — Overlay/present teardown relies entirely on the launcher's shutdown WaitIdle** (LOW-10, INFO-3 overlay decorator nodes, INFO-7 submission fence discards an armed submission on Dispose). Works today because the launcher drains before node disposal; the reliance is undocumented and sibling nodes don't share it. Either document the ordering contract or add local drains.

**T10 — Singles worth individual attention:** MEDIUM-3 `DirectXGpuComputeRecorder` barrier-state cache keyed by raw resource pointer never evicts (pointer reuse inherits dead state); LOW-25 `ViewStack.Register` update-in-place drops the old `IViewContent` undisposed (leaks a whole offscreen engine); LOW-31 `WorldScreenBinder` post-dispose reads return destroyed handles; LOW-32 `CameraFeed` webcam disconnect wedges the camera for process lifetime; LOW-6 `TrackerDispose` leaves a disposed static behind; LOW-7 ceiling-refused camera view cached undisposed; LOW-18 XInput rumble left latched at teardown; LOW-8 `DirectXDeviceContext` unsynchronized lazy init; LOW-20 Wayland `wl_compositor` proxy leak; LOW-22 `Win32D3D11VideoDevice` ignored `CreateQuery` HRESULT can hang the grabber thread; LOW-1 BareMetal sample partial-bring-up leak (experimental tree, lowest priority).

## High severity (2)

### HIGH-1. DirectXSurfaceUpload — `missing-gpu-sync` (batch: directx-b)

DirectXSurfaceUpload's extent-change rebuild path releases the sampled texture with no GPU synchronization, while the frame ring deliberately keeps frames in flight that sample it.

**Evidence:** src/Puck.DirectX/Interop/DirectXSurfaceUpload.cs:265-275 (EnsureResources: on a width/height/format change it calls DisposeImageResources() at :275 with no GPU wait), :432-435 (DisposeImageResources releases m_uploadBuffer and m_texture immediately), :438-452 (Dispose, by contrast, drains the full queue via WaitForGpu before the same release). Consumer reliance on the resize path: src/Puck.SdfVm/Views/CpuSurfaceSource.cs:74-83 (comment: 'recreates its image on a dimension/format change, so a varying capture extent needs no manual reallocation'). The engine's own doctrine that only a device drain proves external consumers idle: src/Puck.Demo/GamingBrickChildNode.cs:1840-1844.

**Failure scenario:** The demo pipelines frames (per-frame WaitIdle was deliberately removed; frame-ring depth 2-3). Frame N's world submit samples the upload's texture as a screen source and is still executing on the GPU when frame N+1 calls Upload() with a different extent — e.g. a WGC window-capture source whose captured window the user resizes mid-run (CpuSurfaceSource explicitly supports 'a varying capture extent'), or a live dmg<->agb device swap changing the machine framebuffer from 160x144 to 240x160. EnsureResources releases the texture that frame N is still sampling: D3D12 use-after-free -> DXGI_ERROR_DEVICE_REMOVED or corruption. The only wait in Upload (WaitForGpu at :217) runs AFTER the release and only covers work submitted before it. Note the sibling VulkanSurfaceUpload has the same gap on its resize path (src/Puck.Vulkan/VulkanSurfaceUpload.cs:238 — DisposeResources waits only its OWN pending copy fence via WaitForPendingUpload, not device idle; its full drain, TryWaitIdle at :340, is Dispose-only).

**Action:** In D:\Source\ByteTerrace\Puck\src\Puck.DirectX\Interop\DirectXSurfaceUpload.cs, at the top of the EnsureResources rebuild path (immediately before the DisposeImageResources() call at line 275), drain the queue first: if ((0 != m_texture) && (0 != m_fence) && (0 != m_deviceContext.CommandQueueHandle)) { WaitForGpu(); }. The fence/event already exist (created in the constructor) and WaitForGpu is the existing private helper at :411. Apply the same fix to the Vulkan peer: in src/Puck.Vulkan/VulkanSurfaceUpload.cs EnsureResources, when the rebuild path fires with an existing image (0 != m_imageViewHandle), call m_device?.TryWaitIdle() before DisposeResources() at :238 (mirroring the Dispose path at :340). The drain only fires on an extent/format/device change, matching the precedent and rationale documented at GamingBrickChildNode.cs:1840-1844.

### HIGH-2. VulkanSurfaceUpload — `missing-gpu-sync` (batch: vulkan-b)

Mid-life rebuild (extent/format/device change) destroys the sampled image and view without draining GPU consumers — only the public Dispose() waits idle.

**Evidence:** src/Puck.Vulkan/VulkanSurfaceUpload.cs:238 (EnsureResources calls DisposeResources on rebuild), :309-327 (DestroyImageView/DestroyColorImage), :291 (WaitForPendingUpload drains ONLY this object's own staging copy, not consumers sampling the view), vs :340 (Dispose() calls m_device?.TryWaitIdle() — the rebuild path has no equivalent). Consumer reality: CpuSurfaceSource.cs:75-76 documents 'recreates its image on a dimension/format change, so a varying capture extent'; SdfWorldEngine submits frames fire-and-forget in a fence ring (SdfWorldEngine.cs:1300), so the previous frame that sampled the current view handle can still be executing when the next Upload rebuilds.

**Failure scenario:** Live demo: a WGC capture/console feed changes extent (user resizes the captured window, cabinet pane relayout) while frame N-1 — which sampled the upload's image view in the SDF views kernel or the presenter blit — is still in flight. EnsureResources destroys that image/view mid-execution: validation error, undefined behavior, potential device-lost on stricter drivers. Public Dispose is safe; only the rebuild path races.

**Action:** In src/Puck.Vulkan/VulkanSurfaceUpload.cs EnsureResources, before the DisposeResources() call at line 238, drain the device when rebuilding an existing set: `if (m_device is not null) { m_device.TryWaitIdle(); }` (mirroring Dispose() at line 340). Rebuilds only fire on extent/format/device changes, so the stall is rare and acceptable.

## Medium severity (7)

### MEDIUM-1. BakePreviewService + BakeRasterizer — `missing-device-lost-reset` (batch: demo-forge)

The bake preview chain is never reset on device loss: after a device rebuild, BakeRasterizer submits GPU work on an SdfWorldEngine wrapping the destroyed device, and the easel keeps sampling a destroyed VkImageView handle.

**Evidence:** OverworldRenderNode.cs:3241-3252 (OnDeviceLost forwards only to m_root, m_bricks, agb service); OverworldRenderNode.cs:3257 (DisposeBakePreview exists only in Dispose); BakeRasterizer.cs:117-122 (EnsureEngine compares only width/height, never device identity, so m_engine built on the lost device is reused); BakePreviewService.cs:59,207 (CurrentImageViewHandle keeps the dead device's image-view handle); OverworldFrameSource.cs:2299 (easel samples that handle every frame while creator is active); contrast AdvancedGamingBrickHost.cs:262-266 (the engine's established device-lost pattern: dispose upload, zero handle); LauncherWindowHostedService.cs:777 (OnDeviceLost is a real, driven recovery path)

**Failure scenario:** Creator mode open with a live easel preview; the GPU device is lost and the host rebuilds (the wired OnDeviceLost/rebuild flow). The next debounced bake calls BeginRasterize -> EnsureEngine sees a matching extent and reuses m_engine whose device handles were destroyed -> UploadProgram/SubmitFramePipelined on freed device objects (access violation or immediate second device loss). Independently, CreatorPreviewHandle still returns the old device's image-view handle, which the new device's descriptor writes consume.

**Action:** In OverworldRenderNode.OnDeviceLost (src/Puck.Demo/Overworld/OverworldRenderNode.cs:3241) add m_frameSource?.DisposeBakePreview(); (DisposeBakePreview at OverworldFrameSource.cs:2381-2385 already nulls m_bakePreview so CreatorPreviewHandle degrades to 0). Then confirm the resource-rebuild path re-runs the build method containing frameSource.InstallBakePreview (OverworldRenderNode.cs:1528) after a device rebuild so the preview reinstalls; if it does not, instead give BakePreviewService an OnDeviceLost that disposes m_rasterizer's engine and m_upload WITHOUT device waits, zeroes CurrentImageViewHandle, and resets m_state to Idle, mirroring AdvancedGamingBrickHost.cs:262-266. Note: plain Dispose routes through TryWaitIdle paths that tolerate a lost device, so DisposeBakePreview is safe to call there.

### MEDIUM-3. DirectXGpuComputeRecorder — `use-after-dispose` (batch: directx-a)

m_imageStates tracks legacy-barrier resource state keyed by raw ID3D12Resource* and never evicts entries when an image is destroyed, so a new texture allocated at a reused pointer inherits the dead image's tracked state.

**Evidence:** src/Puck.DirectX/DirectXGpuComputeRecorder.cs:36 (m_imageStates declaration), :201 (GetOrAdd seed), :220 (state write); no removal/eviction anywhere (grep confirms only these three touches). Consumers destroy storage images mid-session: src/Puck.Demo/GamingBrickChildNode.cs:1517 (m_outputImage per cabinet), SdfWorldEngine.cs:1513 (m_storageImage per engine, released mid-run per the comment at SdfWorldEngine.cs:1457-1462).

**Failure scenario:** On a device without Enhanced Barriers (legacy fallback at DirectXGpuComputeRecorder.cs:198-221, live on a Windows 10 D3D12 runtime), a cabinet is shut down (output image released) and a new cabinet boots; the new UNORDERED_ACCESS-created texture lands at the old resource's address. TransitionImageLayout's GetOrAdd at :201 returns the stale entry (e.g., PIXEL_SHADER_RESOURCE): either the barrier is skipped (before==after at :204) or StateBefore is wrong — a D3D12 validation error and undefined rendering. The recorder is a process-lifetime DI singleton, so entries accumulate across every cabinet/engine created in a long demo session.

**Action:** In the legacy branch of TransitionImageLayout (src/Puck.DirectX/DirectXGpuComputeRecorder.cs:198-221): when oldLayout == GpuImageLayout.Undefined (the image's first transition), overwrite the tracked entry with D3D12_RESOURCE_STATE_UNORDERED_ACCESS (m_imageStates[imageHandle] = UNORDERED_ACCESS, then use it as `before`) instead of GetOrAdd, so a recreated image at a reused address re-seeds correctly. Optionally also remove the entry when the storage image is destroyed.

### MEDIUM-6. VulkanSurfaceReadback — `contract` (batch: vulkan-b)

Blocking Read() ignores an un-mapped SubmitRead: it re-records the still-pending command buffer, and on an extent change DisposeResources destroys the fence and buffer the GPU may still be writing; the guard that prevents this exists on every SdfWorldEngine submit path except ReadPixels.

**Evidence:** src/Puck.Vulkan/VulkanSurfaceReadback.cs:75-118 (Read never checks m_readInFlight; SubmitRead does at :139-141), :307-322 (DisposeResources destroys fence+buffer with no WaitForFence, unlike VulkanSurfaceUpload.cs:291's WaitForPendingUpload). **Action:** Two one-liners: (1) add ThrowIfPipelinedFrameInFlight() at the top of SdfWorldEngine.ReadPixels (src/Puck.SdfVm/SdfWorldEngine.cs:1366). ### MEDIUM-7. VulkanSurfaceUpload — `missing-gpu-sync` (batch: vkpresent-world)

Both surface-upload primitives' extent/format-change rebuild path destroys the previous GPU image without draining consumers: only the upload's own copy fence is waited (Vulkan) or nothing is (D3D12), while the frame ring allows the prior frame — which sampled that image — to still be executing.

**Evidence:** src/Puck.Vulkan/VulkanSurfaceUpload.cs:224-238 (EnsureResources extent/format/device mismatch → DisposeResources) and :287-331 (DisposeResources waits only the upload's OWN copy fence via WaitForPendingUpload, then destroys the image view/image) — contrast Dispose() :334-342 which does m_device?.TryWaitIdle() first. Same shape on D3D12: src/Puck.DirectX/Interop/DirectXSurfaceUpload.cs:265-275 (EnsureResources → DisposeImageResources, no WaitForGpu) vs Dispose :438-452 (WaitForGpu before release). Consumers pipeline: VulkanRenderer.cs:50 PresentFrameRingSize=2 with WaitForFrameSlot :257-267 waiting only the slot fence, so frame N-1 may still sample the old view while frame N's Publish triggers the rebuild.

**Failure scenario:** Latent today: every current producer publishes at a fixed extent (Win32GraphicsCaptureFeed scales to its fixed target extent per src/Puck.Platform/Windows/Win32GraphicsCaptureFeed.cs:215,342; the camera session extent is fixed per open; machine framebuffers are fixed), so EnsureResources' mismatch branch fires only on first use or device change. The first producer whose published extent varies mid-session (a future native-extent capture mode, a resizable creation preview) makes this live: frame N-1's compute/blit still references the old VkImageView/ID3D12Resource on the queue when frame N's publish destroys it — validation error, corruption, or device removal in the long-running interactive process, on both backends.

**Action:** In src/Puck.Vulkan/VulkanSurfaceUpload.cs EnsureResources (line 238), call m_device?.TryWaitIdle() (or an equivalent consumer drain) before DisposeResources when tearing down a previously published image (0 != m_imageViewHandle); in src/Puck.DirectX/Interop/DirectXSurfaceUpload.cs EnsureResources (line 275), call WaitForGpu() before DisposeImageResources() when 0 != m_texture. The rebuild is a rare event, so the full drain is acceptable; alternatively defer-destroy the old image behind the next frame fence.

## Low severity (34)

### LOW-1. Vk (static) / VulkanResource / DisposalScope — `leak` (batch: bare-metal)

Partial Vulkan bring-up failure leaks instance/surface/device (Shutdown never called), and the obvious fix — tracking VulkanResource right after Initialize succeeds — would null-deref because Shutdown's destroy function pointers are loaded only inside InitializeRendering.

**Evidence:** experimental/Puck.BareMetal/samples/VulkanWindow/Program.cs:63-67 (VulkanResource only tracked after BOTH Vk.Initialize and Vk.InitializeRendering succeed; `||` short-circuit returns 2 with no teardown); experimental/Puck.BareMetal/samples/VulkanWindow/Vk.cs:100,114,156,187 (Initialize's own mid-sequence failure returns leave Instance/Surface/Device live with no cleanup); experimental/Puck.BareMetal/samples/VulkanWindow/VkRender.cs:88-118 (all destroy function pointers loaded only in LoadRenderProcs, called only from InitializeRendering at line 148), VkRender.cs:141-143 (Shutdown guards on handle != 0 but never null-checks s_destroyDevice/s_destroySurface/s_destroyInstance; only s_deviceWaitIdle is null-checked at line 124)

**Failure scenario:** vkCreateDevice fails on Vk.cs:187 (or InitializeRendering fails, e.g. vkCreateSwapchainKHR at VkRender.cs:200): Main returns before scope.Track(new VulkanResource()), so Vk.Shutdown() never runs and the already-created VkInstance/VkSurfaceKHR/VkDevice are leaked; the `using` also disposes WindowResource, destroying the HWND while the surface still references it.
